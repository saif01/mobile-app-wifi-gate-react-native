**Prompt for Cursor: Build “WiFiGate” Firewall Login App with Expo React Native**

Build a **production-standard mobile app named `WiFiGate`** using **Expo React Native + TypeScript**. First understand the current real-world firewall login process that users now do manually from a mobile browser, then design and implement the app in a secure, stable, scalable, and maintainable way.

## App Name

**WiFiGate**

## Project Context

Currently, users open this firewall login page in the mobile browser:

`http://10.64.4.253:8090`

Then they enter their **ID** and **Password**. If login is successful, the device can use the WiFi internet.

Now build a mobile app called **WiFiGate** that manages and automates this firewall login flow properly.

---

## Main Requirements

### 1. Editable Firewall Endpoint

* The default firewall login endpoint is:
  `http://10.64.4.253:8090`
* This endpoint must be **editable from inside the app**
* Create a configuration/settings screen where the user can:

  * view the current endpoint
  * edit and save the endpoint
  * validate endpoint format before saving
* Save the endpoint locally and load it on app startup

### 2. WiFi-Based Login Restriction

* Login must be possible **only when the mobile device is connected to an allowed WiFi network**
* Multiple allowed WiFi networks should be supported
* When the app opens:

  * detect the current network connection
  * check whether the device is connected to WiFi
  * compare the current WiFi information with the configured allowed network list
* If matched:

  * allow login
* If not matched:

  * block login
  * show a clear warning message

### 3. Multiple Allowed WiFi Match Logic

* Add support for configuring multiple allowed WiFi entries
* Each entry may contain:

  * SSID / WiFi name
  * gateway IP / local IP match rule / network identifier
  * remarks
  * active/inactive status
* Create reusable logic to match the current connected WiFi with one of the configured entries

### 4. Mobile Data Handling

* When the device is connected to an allowed WiFi, the app should try to ensure the firewall login process uses WiFi
* If Expo cannot directly disable mobile data, implement the best possible practical logic:

  * detect current network type
  * warn user if mobile data may interfere
  * prefer WiFi connection path
  * structure the code so native enhancement can be added later if needed
* Clearly document technical limitations where direct OS-level network disabling is not possible

### 5. Firewall Login Flow

* Create a login screen with:

  * User ID input
  * Password input
  * Show/Hide password
  * Remember credentials option
  * Login button
* On login:

  * verify current WiFi matches allowed list
  * use the saved endpoint
  * submit the ID and Password to the firewall login process
* Handle:

  * successful login
  * invalid credentials
  * timeout
  * endpoint unreachable
  * unauthorized WiFi
  * unexpected response
* On success:

  * mark the user as authenticated
  * show success status
  * navigate to dashboard/home screen

### 6. Stored ID and Password

* User ID and Password should be stored securely
* Never store credentials in plain text
* Use **Expo SecureStore** for credentials and sensitive auth/session data
* Auto-fill saved credentials if the user enables that feature

### 7. Fingerprint Login

* If the mobile device supports fingerprint authentication, enable biometric login
* Use Expo-compatible biometric authentication
* Fingerprint login should only be enabled after at least one successful manual login
* On successful biometric authentication:

  * retrieve saved credentials securely
  * attempt firewall login automatically
* On biometric failure:

  * allow fallback to manual login
* Add setting to enable or disable fingerprint login

### 8. Background / Keep-Alive Behavior

* The app should preserve session state when running in background
* Implement the maximum reliable background-safe behavior Expo allows
* Structure services so future native enhancement can support:

  * periodic revalidation
  * keep-alive
  * silent session checks
* If full background keep-alive is limited by Expo, document that clearly and implement the best possible supported solution

## Important Technical Requirement

Before implementation, first analyze how the firewall login page actually works:

* whether the page uses HTML form POST
* whether cookies/session are required
* whether hidden input fields exist
* whether redirects happen after login
* whether login can be done with direct request submission
* whether WebView-assisted login is more reliable

Then choose the best implementation approach:

1. **Direct HTTP form submission** if reliable
2. **Hybrid/WebView-assisted flow** if the firewall page requires browser session/cookies/hidden form handling

The solution must preserve app-level features such as:

* WiFi checking
* saved credentials
* fingerprint login
* editable endpoint
* status tracking
* logs
